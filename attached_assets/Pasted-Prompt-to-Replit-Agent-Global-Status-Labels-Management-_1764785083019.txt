Prompt to Replit Agent – Global Status Labels Management Tool

Act as a senior full-stack engineer.
I want to centralize all status labels in the system into a single management tool in the Settings page, so the owner can add, rename, or remove statuses without touching the code.

The goal:

All status texts like:
"تم الاعتماد"، "قيد المراجعة"، "تم الشحن"، "تم التوصيل"، "ملغي"…
should be managed from one place in Settings.

The admin should be able to:

Add new statuses

Edit existing statuses (change Arabic label, color, icon)

Delete statuses (if not critical / in use)

All pages (orders, quotes, imports, missing, accounts, etc.) must read their labels from this central configuration instead of hard-coded strings.

1. Internal status keys vs. display labels

Very important:
We must separate internal status keys from display labels.

Define a central type and data structure, for example:

type StatusDomain =
  | 'orders'
  | 'quotes'
  | 'imports'
  | 'accounts'
  | 'missing'
  | 'payments'
  | 'general';

type StatusDefinition = {
  key: string;          // internal key, e.g. "APPROVED", "SHIPPED"
  label: string;        // Arabic UI text, e.g. "تم الاعتماد", "تم الشحن"
  color?: string;       // e.g. 'green', 'blue', 'orange', 'red'
  icon?: string;        // optional icon name
  isDefault?: boolean;  // optional default for this domain
  isSystem?: boolean;   // if true: can't be fully deleted, only renamed
};

type StatusConfig = {
  [domain in StatusDomain]: StatusDefinition[];
};


All business logic (conditions) should use status.key,
while the UI uses status.label from this config.

Store this config in mockApi/localStorage so it can be edited from the Settings UI.

2. Status Management UI in Settings

Create a new section in Settings called:

"إدارة مسميات الحالات"

This page should:

Show a list of domains as tabs or a sidebar:

طلبات (orders)

عروض الأسعار (quotes)

طلبات الاستيراد (imports)

طلبات الحسابات (accounts)

النواقص (missing)

"عام" أو "مشترك" (general)

When a domain is selected, display a table of statuses for that domain:

Columns:

Arabic label (label) – editable

Internal key (key) – read-only or editable with caution

Color (select or color picker)

Icon (optional select from a set)

Default (radio button or toggle)

System (badge to indicate system-protected status)

Provide actions:

"إضافة حالة جديدة":

Opens a small form to add:

key (suggest using uppercase English, e.g. "UNDER_REVIEW")

Arabic label (e.g. "قيد المراجعة")

color

icon (optional)

Edit:

Allow changing the Arabic label, color, icon, default flag.

Optionally allow changing key ONLY if not used in critical logic (or show a big warning).

Delete:

Allow deleting non-system statuses if they are not used on any records.

For system statuses (isSystem = true), either:

disable delete, or

require choosing a replacement status.

Save all changes back to the central StatusConfig structure.

3. Wire all pages to use central statuses

Refactor the places where statuses are used so they no longer use hard-coded Arabic strings.

Identify all modules that have statuses:

Orders

Quotes

Imports

Account opening requests

Missing parts

Any other workflows

Replace hard-coded strings and enums with references to StatusConfig:

For each record (order/quote/import/account/missing), store an internal statusKey like "APPROVED", "PENDING", "SHIPPED", "CANCELLED".

When rendering in UI:

Look up the domain and statusKey in StatusConfig.

Display label and use color/icon as defined there.

For filters and dropdowns (e.g. status filter on a table):

Populate the dropdown options from StatusConfig[domain], not from hard-coded arrays.

This way, if the owner adds a new status in Settings, it automatically appears in filters and forms.

4. Adding, editing, and deleting statuses safely

Implement safety rules:

Adding a new status:

Must not duplicate an existing key in that domain.

The label can be anything (Arabic).

Editing a status:

Changing label, color, icon is always allowed.

Changing key should be:

either forbidden for system statuses

or carefully handled by updating existing records.

Deleting a status:

If it is system-critical (isSystem = true), show a message:

"لا يمكن حذف هذه الحالة لأنها حالة نظامية."

If it is non-system:

Check if any records currently use this status.

If there are records, show options:

"لا يمكن الحذف لأن هناك سجلات مرتبطة"

or "اختر حالة بديلة ليتم تحويل هذه السجلات إليها" (map all to another statusKey)

Default statuses:

For each domain, allow marking one status as default (e.g. for new orders or new account requests).

Enforce that only one default = true per domain.
